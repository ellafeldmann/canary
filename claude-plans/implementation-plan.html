01

Overview

The service ingests structured signals from four independent public sources — GDELT (political/conflict events), FRED (economic stress), USGS (seismic activity), and NOAA (severe weather) — on a recurring schedule, without any always-on server. Each source is normalized to a shared 0–5 severity scale so unlike signals become comparable, and an alert fires the moment a source (or the composite) crosses a threshold.

Loosely modeled on kylemcdonald/ews, which tracks business-jet traffic against a learned weekly/holiday baseline as a single-domain proxy for the same pattern. This plan generalizes that approach — baseline deviation, standard-deviation bands, threshold alerting — across four independent domains instead of one.

02

Goals & non-goals

◆ in scope for v1

  • Ingest and score all four domains on independent schedules, no server to run
  • Every source reduces to the same 0–5 severity scale
  • Push alerts (Telegram) the instant a threshold is crossed — not just visible on a dashboard someone has to check
  • Zero paid infrastructure

○ explicitly deferred

  • UI / dashboard — a published JSON snapshot is enough for now, no server required
  • Realtime / streaming updates — cron cadence (15–30 min) is the floor
  • Multi-recipient / subscriber management — single operator only
  • Forecasting — this flags deviation from baseline, it does not predict

05

Build phases

Ordered so the full pipeline is proven end-to-end on the simplest source before the harder baseline modeling begins.

phase 1USGS — single source, end to end

No API key, and the pre-built GeoJSON feed already ships a computed severity (sig, and PAGER alert color). This proves fetch → store → threshold → Telegram without also having to design a baseline model yet.

Deliverables: sources/usgs.py polling the significant-events feed every 15 min · store.py writing to SQLite · alert.py sending a Telegram message on alert=orange|red or sig above a fixed threshold.

done whenA real earthquake alert reaches Telegram within 15 minutes of USGS publishing it.
phase 2Add GDELT, FRED, NOAA

GDELT — poll the DOC 2.0 API every 30 min; track Goldstein scale and protest-coded event volume per country against a rolling 90-day baseline.

FRED — poll daily; track T10Y2Y, STLFSI4, ICSA; flag by historical percentile rather than a rolling baseline, since macro series don't have a weekly pattern to model.

NOAA — poll /alerts/active every 15 min; count Extreme+Severe alerts nationally against a month-of-year baseline, since severity is seasonal.

done whenAll four sources write readings + severities into the same schema, independently of one another.
phase 3Composite scoring

Map every source onto the shared 0–5 scale (already true for USGS/NOAA by construction; GDELT/FRED need an explicit z-score/percentile → 0–5 mapping). Composite = max(severities), not an average — one domain in crisis shouldn't get diluted by three quiet ones.

done whenOne alert.py entry point can answer "what's our worst active signal right now" across all four sources.
phase 4 · deferredUI, realtime, Postgres

Static snapshot + dashboard (publish JSON to R2 or GitHub Pages — no server). A queryable API for other consumers is a thin Cloudflare Worker reading Turso directly — a small addition, not a migration, since storage is already network-accessible (§06). Realtime push and a Postgres migration are bundled together later, since both only become worth doing once an always-on process exists for some other reason.

06

Data model

Five tables. source_id is a foreign key into a sources lookup rather than a bare string, and readings split by shape: scalar series (FRED, GDELT aggregates, NOAA counts) go in readings; discrete, multi-field events (individual earthquakes, individual NOAA alerts) go in events, keyed so a re-polled event updates instead of duplicating.

decidedTurso (hosted, libSQL/SQLite-compatible) over a git-committed file. A file in git is only reachable by the workflow that commits it — it can't be queried by anything else. Turso is network-accessible from day one, so ingestion (Python, via GitHub Actions) and a future read API (e.g. a Cloudflare Worker) can query the same live database without a storage migration in between. The schema above is the actual contract for future consumers, not any Python code — nothing about it is language-specific.
CREATE TABLE sources (
  id TEXT PRIMARY KEY,          -- 'usgs' | 'gdelt' | 'fred' | 'noaa'
  label TEXT NOT NULL,
  cadence_minutes INTEGER NOT NULL
);

-- scalar time series: FRED series, GDELT daily aggregates, NOAA alert counts
CREATE TABLE readings (
  id INTEGER PRIMARY KEY,
  source_id TEXT NOT NULL REFERENCES sources(id),
  fetched_at TEXT NOT NULL,      -- ISO8601 UTC
  metric TEXT NOT NULL,          -- 'goldstein', 'T10Y2Y', 'extreme_alert_count', ...
  region TEXT,                   -- country/state code, nullable
  raw_value REAL,
  severity REAL,                 -- normalized 0-5
  meta_json TEXT                 -- source-specific extra fields
);

-- discrete events: individual earthquakes, individual NOAA alerts
CREATE TABLE events (
  id INTEGER PRIMARY KEY,
  source_id TEXT NOT NULL REFERENCES sources(id),
  external_id TEXT NOT NULL,     -- USGS event id / NOAA alert id -- dedup key
  occurred_at TEXT NOT NULL,
  region TEXT,
  severity REAL NOT NULL,         -- normalized 0-5
  payload_json TEXT NOT NULL,     -- full record: mag, depth, place, lat/lon, alert color...
  UNIQUE (source_id, external_id)
);

CREATE TABLE baselines (
  source_id TEXT NOT NULL REFERENCES sources(id),
  metric TEXT NOT NULL,
  region TEXT,
  window_start TEXT NOT NULL,
  mean REAL,
  stddev REAL,
  PRIMARY KEY (source_id, metric, region, window_start)
);

CREATE TABLE alerts (
  id INTEGER PRIMARY KEY,
  source_id TEXT NOT NULL REFERENCES sources(id),
  event_id INTEGER REFERENCES events(id),  -- null for readings-based alerts
  triggered_at TEXT NOT NULL,
  severity REAL NOT NULL,
  message TEXT NOT NULL,
  delivered INTEGER DEFAULT 0
);

Found during Phase 1 implementation: without event_id, nothing distinguishes "already alerted for this earthquake" from "alerted for a different one," so a still-active USGS event would re-trigger a message on every 15-minute poll. Nullable, since scalar-series alerts (FRED/GDELT/NOAA) won't reference a specific event.

07

Scoring approach

Every source reduces to one of two shapes, and each shape has its own table (see §06):

  • Continuous series → readings, z-score against a trailing baseline (GDELT tone/volume, FRED series, NOAA alert counts)
  • Event stream with built-in severity → events, direct threshold, no baseline needed (USGS sig / alert)

Baseline windows are source-specific, not one-size-fits-all: GDELT and NOAA get seasonal windows (month-of-year, day-of-week) because both have real cyclical patterns; FRED uses historical percentile instead, since macro series don't cycle the same way.

Shared severity legend, used everywhere downstream of scoring:

0–2 · nominal 3 · elevated 4–5 · critical
08

Alerting

Single channel for v1: a Telegram bot posting to one chat ID (the operator). Behavior is keyed off severity, not just presence of a reading:

SeverityBehavior
0–2Logged only, no message sent
3One Telegram message
4–5Message sent, repeated every N hours until the reading clears baseline

Every crossing writes one row to alerts with a cooldown enforced in config.yml, so a sustained anomaly doesn't produce one message per poll cycle.

09

Observability & failure handling

Distinct from domain alerting (§08): that's "is a signal elevated." This is "is the pipeline itself broken" — a source changes its schema, an API times out, a workflow throws. Two free layers, no new service added.

Logging

Python stdlib logging, leveled, one line per event, written to stdout. GitHub Actions captures stdout/stderr automatically as the per-run log — browsable in the Actions UI, ~90-day retention by default. No log aggregator needed at four small jobs a day.

Failure notification

  • GitHub Actions' native behavior: emails the workflow's actor when a scheduled run fails — zero setup.
  • A failure() step at the end of each workflow pings the same Telegram bot used for domain alerts, prefixed to distinguish it from a real signal:
- name: notify on failure
  if: failure()
  run: python -m core.alert --pipeline-error "${{ github.workflow }} failed"

Deliberately not reaching for a dedicated error tracker (Sentry, etc.) in v1 — a signup, a key to manage, and a dependency for marginal benefit at this volume. Natural upgrade once this stops being something checked by hand.

Open questions

Repo visibility — public or private?

Public keeps Actions minutes free and unlimited; nothing sensitive needs to live in the repo since keys go in Secrets either way. Leaning public, needs confirmation.

?

Composite formula — is max(severities) the right call long-term, or should it eventually weight by domain?

Starting with max for simplicity; revisit once real data is flowing and false-positive rate is known.

?

Alert channel — Telegram only for v1?

Chosen for setup speed. alert.py is written so email can be added later without restructuring the dispatcher.
